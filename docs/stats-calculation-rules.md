## 1. 概述

效率分析页（Stats Page）是 AskOnce 系统的核心数据看板，用于量化展示知识库的使用效果和自动化效率。该页面通过后端 API `/api/stats/overview` 和 `/api/stats/trends` 提供数据，前端在此基础上进行二次计算并渲染可视化图表。

本文档完整记录了每一项指标的**数据来源、计算公式、阈值定义及边界处理逻辑**，供产品经理、研发工程师及数据分析人员参考。

---

## 2. 数据来源

所有指标均基于以下两张数据库表进行计算：

### 2.1 `KnowledgeEntry` — 知识条目表

| 字段 | 说明 |
|------|------|
| `id` | 主键，知识条目唯一标识 |
| 其他字段 | 知识内容、分类、创建时间等 |

每一行代表知识库中的一条知识条目。统计页使用该表计算知识库总量。

### 2.2 `Interaction` — 用户交互/查询记录表

| 字段 | 说明 |
|------|------|
| `id` | 主键，交互记录唯一标识 |
| `confidence` | 浮点数，系统对本次回答的置信度分数（取值范围 0.0 ~ 1.0） |
| `status` | 字符串，交互状态，可选值包括 `confirmed`、`edited`、`pending` 等 |
| `created_at` | 时间戳，交互发生时间（用于趋势图按天聚合） |

每一行代表一次用户查询及系统的响应记录。

## 3. 核心指标计算规则

### 3.1 知识条目总数 `total_knowledge_entries`

| 项目 | 内容 |
|------|------|
| **含义** | 知识库中当前存储的知识条目总数 |
| **公式** | `COUNT(*) FROM KnowledgeEntry` |
| **数据类型** | 整数 |
| **示例** | 知识库有 1,200 条知识 → 返回 `1200` |

### 3.2 总交互次数 `total_interactions`

| 项目 | 内容 |
|------|------|
| **含义** | 系统累计处理的用户查询总次数 |
| **公式** | `COUNT(*) FROM Interaction` |
| **数据类型** | 整数 |
| **示例** | 用户共发起 5,000 次查询 → 返回 `5000` |

### 3.3 已确认次数 `confirmed_count`

| 项目 | 内容 |
|------|------|
| **含义** | 用户明确确认采纳（直接确认或编辑后确认）的交互次数 |
| **公式** | `COUNT(*) FROM Interaction WHERE status IN ('confirmed', 'edited')` |
| **数据类型** | 整数 |
| **说明** | 包含两种状态：`confirmed`（直接采纳）和 `edited`（编辑后采纳），两者均视为有效采纳 |

### 3.4 采纳率 `adoption_rate`

| 项目 | 内容 |
|------|------|
| **含义** | 用户对系统回答的采纳比例，反映回答质量 |
| **公式** | `ROUND(confirmed_count / total_interactions × 100, 1)` |
| **数据类型** | 浮点数，保留 1 位小数，单位为百分比 (%) |
| **特殊处理** | 当 `total_interactions = 0` 时，返回 `0.0` 以避免除零错误 |
| **示例** | 3,500 次确认 / 5,000 次交互 = `70.0%` |

### 3.5 平均置信度 `avg_confidence`

| 项目 | 内容 |
|------|------|
| **含义** | 所有交互中系统置信度得分的算术平均值 |
| **公式** | `ROUND(AVG(confidence) FROM Interaction, 2)` |
| **数据类型** | 浮点数，保留 2 位小数，取值范围 0.00 ~ 1.00 |
| **特殊处理** | 当 `total_interactions = 0` 时，返回 `0.00` |
| **示例** | 所有查询的置信度平均值为 0.78 → 返回 `0.78` |

### 3.6 自动回复次数 `auto_reply_count`

| 项目 | 内容 |
|------|------|
| **含义** | 置信度达到自动回复阈值，系统可直接发送回答的交互次数 |
| **公式** | `COUNT(*) FROM Interaction WHERE confidence >= 0.90` |
| **数据类型** | 整数 |
| **阈值** | `confidence >= 0.90` |

### 3.7 草稿回复次数 `draft_reply_count`

| 项目 | 内容 |
|------|------|
| **含义** | 置信度处于中等水平，系统生成草稿供人工审核的交互次数 |
| **公式** | `COUNT(*) FROM Interaction WHERE confidence >= 0.60 AND confidence < 0.90` |
| **数据类型** | 整数 |
| **阈值** | `0.60 <= confidence < 0.90` |

### 3.8 低置信度次数 `low_confidence_count`

| 项目 | 内容 |
|------|------|
| **含义** | 置信度低于阈值，需要人工介入处理的交互次数 |
| **公式** | `COUNT(*) FROM Interaction WHERE confidence < 0.60` |
| **数据类型** | 整数 |
| **阈值** | `confidence < 0.60` |

### 3.9 指标一致性校验

以下恒等关系必须成立，可用于数据校验：

```
auto_reply_count + draft_reply_count + low_confidence_count = total_interactions
```

## 4. 置信度阈值说明

系统根据模型返回的置信度分数，将每次交互划分为三个等级：

| 等级 | 置信度范围 | 英文标签 | 系统行为 | 说明 |
|------|-----------|---------|---------|------|
| 🟢 高置信度 | `>= 0.90` | Auto Reply | 自动发送回复 | 系统高度确信答案正确，无需人工审核即可直接回复用户 |
| 🟡 中置信度 | `0.60 ~ 0.89` | Draft Reply | 生成草稿待审核 | 系统有一定把握但不够充分，生成回复草稿交由人工审核确认 |
| 🔴 低置信度 | `< 0.60` | Low Confidence | 需人工介入 | 系统无法给出可靠回答，需要客服人员或专家从头处理 |

**阈值边界处理**：

- 置信度 `= 0.90` 归入**高置信度**（Auto Reply）
- 置信度 `= 0.60` 归入**中置信度**（Draft Reply）
- 阈值均为**左闭右开**区间，高置信度为左闭无上界：`[0.90, 1.0]`、`[0.60, 0.90)`、`[0.00, 0.60)`

## 5. 效率节省时间计算

### 5.1 计算公式

前端函数 `estimateTimeSavedMinutes(stats)` 计算公式如下：

```
节省总时间（分钟） = auto_reply_count × 3 + confirmed_count × 1.5
```

| 场景 | 每次节省时间 | 假设依据 |
|------|------------|---------|
| 自动回复 (`auto_reply_count`) | **3 分钟** | 高置信度回答无需人工审核，相当于完全替代了人工查找、组织和发送回复的全部时间 |
| 已确认草稿 (`confirmed_count`) | **1.5 分钟** | 人工仍需审核但无需从零开始撰写，节省了约一半的处理时间 |

> **注意**: 草稿未确认（`draft_reply_count` 中未被确认的部分）和低置信度交互（`low_confidence_count`）**不计入**节省时间，因为这些场景下人工仍需完整处理。

### 5.2 显示格式

| 条件 | 显示格式 | 示例 |
|------|---------|------|
| 节省时间 < 60 分钟 | `Xm` | `45m` |
| 节省时间 >= 60 分钟 | `Xh Ym` | `2h 30m` |

**格式转换逻辑**：

```
hours = FLOOR(totalMinutes / 60)
minutes = totalMinutes % 60
```

### 5.3 计算示例

假设：`auto_reply_count = 200`, `confirmed_count = 150`

```
节省时间 = 200 × 3 + 150 × 1.5 = 600 + 225 = 825 分钟
显示为: 13h 45m
```

## 6. 知识覆盖率

### 6.1 计算公式

前端函数 `coveragePercent(stats)` 计算公式如下：

```
知识覆盖率 (%) = MIN( ROUND( (auto_reply_count + confirmed_count) / total_interactions × 100 ), 100 )
```

### 6.2 公式解读

| 项目 | 说明 |
|------|------|
| **分子** | `auto_reply_count + confirmed_count` — 被系统成功回答的查询数（自动回复 + 人工确认的草稿） |
| **分母** | `total_interactions` — 所有用户查询总数 |
| **上限** | 取 `MIN(..., 100)` 确保结果不超过 100% |
| **含义** | 衡量知识库对用户问题的回答覆盖程度。覆盖率越高，说明知识库越完善 |

### 6.3 特殊处理

- 当 `total_interactions = 0` 时，返回 `0` 以避免除零错误
- 理论上覆盖率不应超过 100%，但为防御性编程加了 `MIN(..., 100)` 上限

### 6.4 计算示例

假设：`auto_reply_count = 200`, `confirmed_count = 150`, `total_interactions = 500`

```
覆盖率 = MIN( ROUND( (200 + 150) / 500 × 100 ), 100 ) = MIN(70, 100) = 70%
```

## 7. 活动日志

活动日志（Activity Log）以列表形式展示各类交互的统计摘要：

| 日志项 | 显示内容 | 节省时间计算 | 说明 |
|--------|---------|------------|------|
| 🟢 Auto Replied | 数量 + 节省时间 | `auto_reply_count × 3` 分钟 | 高置信度自动回复的次数，展示其带来的时间节省 |
| ✅ Confirmed Drafts | 数量 + 节省时间 | `confirmed_count × 1.5` 分钟 | 人工确认采纳的草稿次数，展示审核模式下的时间节省 |
| 📝 Drafts Generated | 仅数量 | — | 系统生成的草稿总数（`draft_reply_count`），不计算节省时间 |
| 🔴 Low Confidence | 仅数量 | — | 低置信度需人工处理的次数（`low_confidence_count`），不计算节省时间 |

**各项含义说明**：

- **Auto Replied** — 体现系统全自动处理能力，是效率提升的核心指标
- **Confirmed Drafts** — 体现人机协作效率，表明系统提供了有效辅助
- **Drafts Generated** — 反映中置信度场景的数量规模，不直接等同于效率提升
- **Low Confidence** — 反映知识库的短板区域，可作为知识库优化方向的参考

## 8. 趋势图表

### 8.1 数据接口

**API**: `GET /api/stats/trends`

**返回数据结构**: 最近 7 天的每日统计，每天包含以下字段：

| 字段 | 类型 | 说明 |
|------|------|------|
| `date` | 字符串 (YYYY-MM-DD) | 日期 |
| `total` | 整数 | 当日总交互次数 |
| `confirmed` | 整数 | 当日已确认交互次数（status IN ('confirmed', 'edited')） |

### 8.2 聚合逻辑

- **时间范围**: 从当前日期往前推 7 天（含今天）
- **聚合维度**: 按 `created_at` 字段的日期部分分组
- **排序**: 按日期升序排列

### 8.3 图表用途

趋势图表用于展示：

1. **每日交互量变化** — 观察用户活跃度趋势
2. **每日确认量变化** — 观察回答质量趋势
3. **两者对比** — 直观反映采纳率的日间波动

## 9. 注意事项

### 9.1 边界情况

| 场景 | 处理方式 |
|------|---------|
| 知识库为空（`total_knowledge_entries = 0`） | 正常返回 0，前端正常展示 |
| 无任何交互（`total_interactions = 0`） | 采纳率返回 `0.0`，平均置信度返回 `0.00`，覆盖率返回 `0`，节省时间返回 `0m` |
| 所有交互均为高置信度 | `auto_reply_count = total_interactions`，覆盖率可能接近或等于 100% |
| 所有交互均为低置信度 | 节省时间为 0，覆盖率为 0%（如无确认） |

### 9.2 数据一致性

- `auto_reply_count + draft_reply_count + low_confidence_count` 必须等于 `total_interactions`
- `confirmed_count` 是按 `status` 字段筛选的，与按 `confidence` 分档的三个计数互相独立，不存在直接的加和关系
- `confirmed_count` 可能来自任意置信度分档的交互（高、中、低置信度的交互均可能被用户确认）

### 9.3 精度与舍入

| 指标 | 精度 | 舍入方式 |
|------|------|---------|
| `adoption_rate` | 1 位小数 | 四舍五入 |
| `avg_confidence` | 2 位小数 | 四舍五入 |
| `coveragePercent` | 整数 | `Math.round()` |
| 节省时间（分钟） | 整数级显示 | 按格式化规则处理 |

### 9.4 性能考虑

- 概览接口 `/api/stats/overview` 执行全表聚合查询，数据量大时建议添加缓存或物化视图
- 趋势接口 `/api/stats/trends` 仅查询最近 7 天，查询范围有限，性能影响较小
- 前端二次计算（节省时间、覆盖率）为纯内存运算，无性能瓶颈

### 9.5 未来优化方向

- 节省时间的估算系数（3 分钟、1.5 分钟）目前为固定值，后续可根据实际运营数据动态调整
- 置信度阈值（0.90、0.60）目前为硬编码，后续可考虑支持管理后台配置
- 趋势图目前固定为 7 天，后续可扩展支持 14 天、30 天等自定义时间范围
